**Project:** VulnRap (REMEDiSSecurity vulnerability report validation platform)
**Date filed:** 2026-04-30
**Filer:** Project owner

---

## The core complaint

**The Replit platform autonomously generated, queued, and executed
work I never requested — and charged me for it.** The same platform
mechanisms that generated this work also caused repeated rework
cycles (rebase conflicts in generated files, test runs polluting
shipped data files, duplicate proposals from parallel agents) that
multiplied the bill further.

I cannot tell from inside the product which queued work is
necessary, which is duplicative, and which will break my running
application if it runs. There is no preview of cost. There is no
bulk way to stop the queue. The platform keeps adding to it on its
own.

This is a Replit platform problem, not a user-configuration problem.
Replit needs to take responsibility for it.

## 2. Why this is a Replit platform problem

These are five platform mechanisms that, by design, generate billable
rework that I did not request:

### 2.1 — Mandatory follow-up generation

Every task agent that finishes work is **required by Replit's
platform-installed `follow-up-tasks` skill** to call
`proposeFollowUpTasks`. The skill runs whether or not there is a
real gap. With ~15 task agents finishing in a session, each emitting
1–3 drafts, the queue compounds. The user never opted into this
behavior; it ships in the platform.

### 2.2 — No coordination between parallel agents

When Replit runs two task agents in parallel and they finish at the
same time, both can independently propose the same follow-up. Live
example currently in my queue:

- `#571` "Attribute bulk re-arm actions to the reviewer in the audit log"
- `#574` "Attribute bulk re-arm actions in the audit log too"

These are literal duplicates by two parallel agents. The platform
showed both to me as separate billable proposals. If I accept both,
I pay twice.

### 2.3 — Generated-file rebase conflicts charged to me

Replit's task-isolation model puts each task agent on its own branch
and merges sequentially. When two in-flight tasks both touch the
OpenAPI spec, the autogenerated client (`lib/api-zod/src/generated/api.ts`)
conflicts on rebase. The platform makes the second agent re-run
`pnpm codegen`, re-validate, re-test, re-architect-review, and
re-post-merge — billed to me — to resolve a conflict that Replit's
own merge ordering caused.

This happened to **#282, #283, and #397** in this session. Three
billable codegen reruns for one feature.

### 2.4 — E2E tests pollute shipped data files; revert is billed to me

Replit's e2e harness runs the API server against shipped data files
in `artifacts/api-server/data/`. Test runs write real per-IP
rate-limit counters and notification queues into those files, and
the auto-merge bot then commits them. Two separate task agents had
to revert the same kind of pollution:

- `#407` reverted `data/avri-drift-notifications.json`
- `#457` reverted `data/calibration-auth-brute-force-state.json`
  AND added per-test `os.tmpdir()` redirection so future runs stop
  doing it

I paid for the same revert twice because the platform kept causing
the pollution.

### 2.5 — Pre-existing flakes investigated by every task agent

Three failing tests in `artifacts/vulnrap/src/pages/feedback-analytics-auth.test.tsx`
are pre-existing flakes. Every task agent that runs the test suite
hits them, then **must spend agent time** investigating whether its
own change caused them, confirming they reproduce on `main`, and
documenting the conclusion in the merge commit message.

The phrase *"The 3 pre-existing failures in `feedback-analytics-auth.test.tsx`
reproduce on `main` without my changes"* (or close variants) appears
in **nearly every recent merge commit message** in this project.
With ~15 merges in 24 hours, that is roughly **15× duplicated
investigation cost** on three flakes that nobody can prioritize
fixing because the queue won't drain.

## 3. I cannot tell which work is necessary or safe

The platform shows me 134 queued tasks but gives me no way to answer
the questions that actually matter:

- Which of these will break the running app if they run?
- Which are duplicates of work already done?
- Which are duplicates of each other?
- Which are necessary vs. nice-to-have?
- How much will each one cost in agent time before it runs?

There is no per-task cost preview. There is no diff preview. There
is no "this overlaps with #X" warning. There is no risk indicator.
I am being asked to either click "approve" without information or
click "cancel" 134 times.

## 4. I have no way to stop the queue

Confirmed live during this session. I asked the main agent to
cancel two PROPOSED follow-up drafts on my behalf:

```
markFollowUpTaskObsolete({ taskRef: "#593" })
  → "Task #593 is not a follow-up of the current task.
     Only follow-up tasks proposed from this task can be marked
     obsolete."

markFollowUpTaskObsolete({ taskRef: "#586" })
  → same error
```

The cancellation API is restricted to follow-ups proposed by the
**currently running** agent. Since the agents that proposed all 124
new tasks have already exited, **nothing in the system can cancel
them programmatically**. Not the main agent, not the user CLI, not
any automation I can write.

The only ways to clear the queue are:

1. Click each of 134 tasks individually in the task panel UI, or
2. Let the platform run all of them and pay for each.

There is no "cancel all proposed", no "cancel all matching
pattern", no batch-cancel button anywhere in the product.

## 7. What Replit needs to do

Because these are platform mechanisms, only Replit can fix them.

### 7.1 — Take responsibility for platform-generated rework

I should not be paying for:

- Codegen rebase reruns that exist solely because of Replit's
  parallel-branch merge ordering
- Reverts of test pollution caused by Replit's e2e harness writing
  to shipped data files
- Per-page rollouts of features that should have been one
  cross-page task
- Duplicate proposals from parallel agents that the platform
  failed to dedupe
- Investigation of the same three baseline flakes by every agent
  in every merge

Please review my project-task billing for the past 30 days and
credit the duplicated and platform-induced runs.

### 7.2 — Fix the platform to prevent this happening again

1. **Bulk-cancel for project tasks.** "Cancel all PROPOSED",
   "cancel all PENDING", or "cancel matching pattern". Today the
   only path is one-click-at-a-time for 134 items.
2. **Disable or opt-in auto-proposed follow-ups.** Make
   `proposeFollowUpTasks` user-opt-in instead of skill-mandated,
   or add a project-level kill switch.
3. **Per-task cost preview** before the queue runs, with overlap
   warnings ("this proposal touches the same files as `#X`").
4. **Risk indicator** on each queued task: "this will modify your
   running production code" vs "this is test-only" vs
   "this is documentation."
5. **Dedupe parallel proposals** before they reach the queue.
   `#571` and `#574` are obviously the same.
6. **Pre-commit guard** against committing changes to
   `artifacts/api-server/data/*.json` (and similar runtime-state
   directories) so the test-pollution → revert cycle stops.
7. **"Freeze the queue" mode** where in-flight tasks finish but no
   new follow-ups are auto-accepted until I un-freeze.
8. **Allow the main agent to cancel arbitrary tasks** in its own
   project, not just its own follow-ups.

### 7.3 — Give me visibility I don't have today

- A billing breakdown per task agent run, per project, per day.
- A clear distinction in the task panel between "user-requested"
  and "platform-auto-proposed" tasks.
- An export of what the platform did on my behalf in the last
  N hours, so I can audit it.
