## 2. What the user cannot do (and why this is the core issue)

The owner of the project has **no programmatic way to mass-cancel** the
auto-proposed tasks. The main agent has the same limitation:

```
markFollowUpTaskObsolete({ taskRef: "#593" })
→ "Task #593 is not a follow-up of the current task.
   Only follow-up tasks proposed from this task can be marked obsolete."
```

This was tested live against `#593` and `#586` during this session.
Both refused. The main agent can only cancel follow-ups that **its own
current task** proposed — which is essentially never, because the main
agent rarely creates follow-up chains. All 120 active tasks were
proposed by other agents that have since finished, so the owning
agent is gone and nobody can cancel from the API.

The **only** way to clear the backlog is:
- Click each task one-by-one in the task panel UI, or
- Wait for the platform to schedule and run all 120, paying for each.

There is no "cancel all proposed", no "cancel all matching pattern",
no batch-cancel API exposed to either the main agent or the user CLI.

## 7. What we are asking Replit support for

1. **A bulk-cancel control for project tasks.** "Cancel all PROPOSED",
   "cancel all PENDING", or "cancel matching pattern". Today the only
   path is one-click-at-a-time in the task panel for 120 items.
2. **An option to disable auto-proposed follow-ups**, or to require
   explicit user opt-in before any `proposeFollowUpTasks` lands in
   the queue.
3. **A per-task-agent cost dashboard** so the user can see spend
   breakdowns per task and per session, before the queue runs.
4. **A guard against parallel agents proposing duplicates.** The
   platform should detect that #571 and #574 are the same proposal
   and dedupe before showing them to the user.
5. **A platform-level pre-commit guard** against committing changes
   to `artifacts/api-server/data/*.json` (and similar runtime-state
   directories) so the test-pollution → revert cycle stops.
6. **Permission for the main agent to cancel arbitrary tasks**
   (not just its own follow-ups), or at minimum, expose a safe
   user-CLI command for batch cancellation.
7. **A "freeze the queue" mode** where in-flight tasks finish but new
   follow-ups are not auto-accepted until the user explicitly
   un-freezes.

## 8. What I will do differently going forward (committed)

1. **Stop auto-proposing parity follow-ups.** When I finish work, I

4. **Refuse to investigate the three baseline flakes** in every

5. **Stop committing to `artifacts/api-server/data/*.json` unless
   the user explicitly asks** — every test now redirects to
   `os.tmpdir()` per the Task #457 fix.
