# Conference Talk: Detecting AI-Generated Vulnerability Reports at Scale

> Reusable submission material for SecurityCon, BSides, OWASP chapters, DEF CON, and similar events.

## 150-Word Abstract

Bug bounty programs are drowning in AI-generated vulnerability reports. Researchers armed with ChatGPT and Claude can now produce convincing-looking advisories in seconds — complete with fabricated reproduction steps, hallucinated CVEs, and plausible-sounding severity assessments. VulnRap is an open-methodology scoring engine that separates signal from slop. It combines five parallel detection strategies — linguistic entropy analysis, technical substance density measurement, CWE coherence verification, adversarial template fingerprinting, and reputation signals — into a single calibrated 0–100 score. Every verdict is fully auditable: reviewers see the exact phrases, weights, and engine outputs that drove the result. In this talk we share the architecture, the detection heuristics we built, the evasion techniques attackers have already tried, and the accuracy metrics from scoring thousands of real-world submissions. You will leave with a concrete playbook for defending your own triage pipeline against AI-generated noise.

## 350-Word Extended Description

Security teams running bug bounty and vulnerability disclosure programs face a new operational threat: high-volume, low-effort reports generated by large language models. These submissions consume triage hours, inflate metrics, and erode trust between programs and legitimate researchers. Existing defenses — rate limits, simple keyword filters, manual review — do not scale.

VulnRap tackles this problem with a multi-engine scoring pipeline that mirrors how an experienced triager reads a report. The Linguistic Analyzer measures bigram entropy, burstiness, and function-word rates to detect the unnatural uniformity of machine-authored prose. The Technical Substance Analyzer counts concrete artifacts — file paths, HTTP headers, stack traces, proof-of-concept payloads — against hedging language and buzzword density. The CWE Coherence engine embeds the report body and verifies that the claimed vulnerability category actually matches the technical content described. The Adversarial Vulnerability Report Index (AVRI) tracks structural templates across submissions and applies campaign penalties when the same skeleton appears repeatedly. Finally, reputation signals flag hallucinated CVE references, fabricated vendor names, and severity inflation patterns.

These five engines feed into an isotonic-calibrated combiner that applies per-CWE weights and override rules. The result is a single score from 0 (almost certainly fabricated) to 100 (high substance, likely genuine). Every score is transparent: the platform surfaces verbatim quotes, individual signal weights, and engine-level breakdowns so reviewers can audit any verdict.

We will walk through real examples of reports that fooled keyword filters but were caught by entropy analysis, reports that contained plausible prose but zero reproducible artifacts, and coordinated submission campaigns detected through template fingerprinting. We will also discuss the adversarial cat-and-mouse: how submitters have attempted to evade detection by injecting real code snippets, randomizing sentence structure, and splitting reports across multiple submissions.

Attendees will leave with: (1) a mental model for the signal categories that distinguish genuine from fabricated reports, (2) practical detection heuristics they can implement in their own triage workflows, and (3) accuracy benchmarks from scoring thousands of submissions across multiple bug bounty platforms.

## Sample Audience Questions + Prepared Answers

**Q1: "Won't publishing your detection methodology make it easier for attackers to evade it?"**

A1: This is the transparency paradox, and we have thought about it carefully. Our experience is that obscurity provides only temporary protection — determined attackers reverse-engineer detection boundaries through trial and error regardless. By publishing our methodology, we gain three advantages that outweigh the risk: (1) the security community can audit our approach and identify blind spots we missed, (2) defenders at other organizations can implement similar heuristics without starting from scratch, and (3) we can iterate faster because we are not spending effort protecting the methodology itself. Our AVRI engine is specifically designed to adapt: when attackers modify templates to evade fingerprinting, the drift-tracking system detects the new patterns and updates its baselines. The arms race is real, but we believe the defender community benefits more from shared knowledge than attackers do.

**Q2: "What is your false-positive rate? How do you handle legitimate reports that happen to use formal or template-like language?"**

A2: On our held-out evaluation set, we measure a 93.7% overall accuracy rate. False positives — genuine reports scored as likely AI-generated — are the failure mode we worry about most, because they risk penalizing legitimate researchers. Several design choices mitigate this. First, no single engine can drive a low score on its own; the combiner requires convergence across multiple signals. A report with formal language but strong technical substance (real file paths, working PoC code, specific version numbers) will score well on the substance engine even if the linguistic engine flags its prose style. Second, the CWE coherence check rewards reports whose technical content matches their claimed category, regardless of writing style. Third, every score is fully auditable — if a researcher disputes a verdict, the reviewer can inspect exactly which signals fired and override the score with documented rationale. We actively monitor false-positive cases to retrain our calibration.

**Q3: "How does this handle reports written in languages other than English?"**

A3: Today, the linguistic analyzer is calibrated primarily for English-language reports, which represent the vast majority of submissions on the platforms we work with. The technical substance engine and CWE coherence engine are largely language-agnostic — they look for code artifacts, file paths, and structural patterns rather than natural-language prose quality. For non-English reports, the linguistic engine's contribution to the final score is automatically down-weighted by the combiner when its confidence is low. We are actively working on multilingual support, starting with the languages most common in bug bounty submissions. The important point is that the multi-engine architecture means no single language-dependent engine can dominate the final score.
